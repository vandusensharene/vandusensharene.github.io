About This Build

Why these metricsI work with sales and booking data every day, so I built this around the numbers I actually watch: revenue against plan, booking ratio, and open pipeline. Each one has a defined formula in the code — nothing is hard-coded.
How the data worksThe page generates its own sample dataset — one row per customer per month, ten fictional accounts across three regions. It's seeded so the numbers don't change between loads, which makes it easier to demo.
The filtersRegion, team, and time period all re-run the aggregation from the raw rows, so every KPI, chart, and table stays in sync. Same drill-down idea I use in my Tableau work, just done in JavaScript.
Booking ratio logicThis is a 3-month rolling average of bookings over sales. Over 1.00 means orders are coming in faster than product is shipping — that's the number I'd flag first in a weekly review.
Chart choicesBars where I'm comparing, lines where I'm trending, horizontal bars where I'm ranking. The detail table is there because summary charts always generate the "okay, but show me the accounts" question.
What it's notNone of this is real customer or employer data — every account name and number is made up. The point is the build, not the dataset.
Sample dataset, fictional accounts. Happy to walk through any part of the code — Sharene Van Dusen, Vantage Operations LLC.
